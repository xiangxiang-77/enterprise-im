# 多智能体协作 Skill

## 适用场景
当任务规模大、涉及多个技术栈/模块、可并行化时，使用 Agent 工具并行委派任务。

## Agent 类型选择

| 场景 | Agent 类型 | 能力 |
|------|-----------|------|
| 广泛代码查询 | Explore | grep + glob + read 只读搜索 |
| 跨文件实现 | general-purpose (claude) | 全工具，写代码 |
| 架构设计 | Plan | 分析 + 设计文档 |
| 代码审查 | claude | 独立审计 |

## 并行策略

### 按模块拆分
```
任务: 实现三个端的功能
  ├── Agent 1: 后端 API + DB migration (Java, SQL)
  ├── Agent 2: 前端页面 (React, TypeScript)
  ├── Agent 3: Qt 桌面端 (C++, Qt 5.9)
  └── Agent 4: Flutter 移动端 (Dart)
```
四个 Agent 同时运行，互不干扰。

### 按层次拆分
```
任务: 补齐后台管理功能
  ├── Agent 1: Controller 层 (REST API)
  ├── Agent 2: Service 层 (业务逻辑)
  └── Agent 3: DB Migration (数据表)
```
注意：分层拆分时 Agent 2/3 先完成，Agent 1 依赖它们的接口定义。

## Agent 提示词模板

### 结构化提示词
```markdown
## Context
- 项目: 企业 IM 系统
- 技术栈: [具体技术栈]
- 当前状态: [已有代码/接口]

## Task
[具体、可验证的任务描述]

## Files to modify
- `path/to/file1`: [改动说明]
- `path/to/file2`: [改动说明]

## Constraints
- [版本限制，如 Java 8, Qt 5.9]
- [编码规范，如 JdbcTemplate 非 JPA]
- [禁止事项，如 不可破坏已有测试]

## Verification
- 运行 `[命令]` 应得到 `[期望结果]`
```

### 反例 (禁止)
```
❌ "基于你的分析，去实现XX功能"  // 太模糊，Agent 无上下文
❌ "修复所有编译错误"            // 范围不明确
❌ "参考已有代码完善XX"          // 没有具体文件路径
```

### 正例 (推荐)
```
✅ "在 AdminOrganizationController.java 中新增 GET /api/admin/departments/tree
   端点，返回递归树形 JSON。使用 jdbcTemplate.query 查询 departments 表，
   递归构建 DepartmentTreeNode(id, name, parentId, children)。
   验证: mvn test -Dtest=AdminApiTest"
```

## 实际案例: 四类缺口补齐 (89项)

```
Phase 1: Qt 桌面端 (30项) → 主会话直接编辑 MainWindow.cpp/.h
Phase 2: Flutter 移动端 (28项)
  ├── Agent A: main.dart (导航/主题/引导页/网络状态)
  ├── Agent B: message_bubble.dart (URL/电话/阅后即焚/系统消息/@高亮)
  └── Agent C: contacts/profile/settings screens (字母索引/名片/DND)
Phase 3: 后台管理 (30项)
  ├── Agent D: AdminOverviewController + WebSocket + Dashboard
  ├── Agent E: AdminOrganizationController + DepartmentDetail
  └── Agent F: AdminAdvancedController + AppVersionController + AdminAdvanced
Phase 4: 外部 Provider (5个)
  ├── Agent G: SMS (Aliyun/Tencent/Demo) + SSO OIDC
  └── Agent H: Push (FCM/APNs) + OCR (Baidu) + Office (OnlyOffice)
Phase 5: 全量验证 → 主会话运行 verify.ps1
```

## 合并注意事项

### Agent 输出检查
1. 检查 Agent 是否修改了意外的文件
2. 检查新增代码是否符合项目规范
3. 检查是否引入了编译/类型错误
4. 运行对应的验证命令

### 常见问题
- 缺失 `#include` / `import` — Agent 生成的代码可能缺少头文件
- Java 版本兼容 — Agent 可能使用 Java 10+ 语法 (`var`)
- 类型不匹配 — Agent 可能使用不存在的方法/属性名
- 全局状态冲突 — 多个 Agent 修改同一文件时需手动合并

### 冲突解决
- 优先保留对现有测试影响最小的版本
- 合并后立即运行完整测试套件
- 不要假设 Agent 输出可直接使用——始终验证
